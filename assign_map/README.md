# 안전점검 인원 배정 지도 자동화

---

### 프로젝트 개요

`assign_map`은 안전점검 원본 엑셀 파일을 업로드하면 주소를 정제하고, 지오코딩 좌표를 생성한 뒤, 서비스처리센터·월별 점검 대상을 지도 위에서 인원별로 분할 배정하는 업무 자동화 프로젝트입니다.

기존에는 담당자가 엑셀 주소 목록을 직접 확인하고 지도에서 위치를 대조하면서 인원별 담당 구역을 나눠야 했습니다. 이 과정은 주소 수가 많아질수록 시간이 오래 걸리고, 오더 수 균형과 실제 이동 동선을 동시에 고려하기 어렵다는 문제가 있었습니다.

본 프로젝트는 다음 과정을 하나의 로컬 웹 애플리케이션으로 통합합니다.

1. 월별 안전점검 엑셀 파일 업로드
2. 단독주택 점검 대상 필터링
3. 주소 단위 집계 및 지오코딩
4. 지도 시각화
5. 인원 수 기준 자동 배정
6. 배정 결과 확인 및 엑셀 다운로드

## 분석 목적 및 기대 효과

### 분석 목적

- 안전점검 대상 주소를 월·서비스처리센터 단위로 표준화한다.
- 주소별 오더 수, 세대 수, 용도, 설치 유형 등을 집계한다.
- 좌표 기반 지도에서 점검 대상 분포를 시각적으로 확인한다.
- 인원 수에 따라 점검 물량을 균등하게 분할한다.
- 담당자별 배정 구역이 지리적으로 지나치게 분산되지 않도록 보조한다.

### 기대 효과

- 수작업 주소 분류와 지도 대조 시간을 줄인다.
- 담당자별 오더 수 편차를 줄여 업무량을 균형화한다.
- 배정 결과를 지도에서 즉시 검토할 수 있어 조정 비용을 낮춘다.
- 월별 데이터 업로드만으로 최신 지도 데이터가 갱신된다.
- 지오코딩 결과를 캐시하여 반복 API 호출 비용과 시간을 줄인다.

## 분석 알고리즘

### 1. 주소 정제 및 집계

엑셀 원본 데이터에서 용도가 `단독주택`으로 시작하는 행만 필터링합니다. 이후 월, 센터, 주소 기준으로 데이터를 그룹화하여 주소별 오더 수와 세대 수를 계산합니다.

주소별로 다음 정보를 생성합니다.

- 오더 수
- 세대 수
- 용도별 건수
- 설치유형별 건수
- 특정유형별 건수
- 원본 주소
- 지번/도로명/좌표 정보

### 2. 지오코딩 및 좌표 보정

좌표 생성은 우선순위를 둡니다.

1. `naver_geocode`: Naver Geocoding API로 주소 직접 매칭
2. `naver_geocode_base_match`: 지번 기본 주소 단위 매칭
3. `dong_centroid_estimate`: 동 중심점 기반 추정 좌표

이미 성공한 주소는 `geocodes/geocoding.json`에 캐시되며, 다음 업로드 시 캐시에 없는 주소만 추가 지오코딩합니다. 이를 통해 반복 실행 시 처리 시간을 줄이고 API 호출량을 절감합니다.

### 3. 블록 및 인접 관계 구성

좌표가 생성된 주소들은 거리 기준으로 가까운 주소끼리 인접 관계를 계산합니다.

주요 기준:

- 지구 반경 기반 Haversine 거리 계산
- 최대 인접 거리 기준
- 주소별 최대 이웃 개수 제한
- 일정 규모 이상 블록 분할

이 정보는 지도 표시와 배정 품질 보조에 사용됩니다.

### 4. 인원 배정 분할 알고리즘

현재 인원 배정은 정렬된 주소 목록을 인원 수만큼 연속 구간으로 나누는 동적 계획법(Dynamic Programming) 기반 방식입니다.

#### 기본 아이디어

전체 오더 수를 인원 수로 나누어 1인당 목표 오더 수를 계산합니다.

```text
target = 전체 오더 수 / 인원 수
```

그리고 주소 목록을 동, 지번, 도로명 기준으로 정렬한 뒤, 각 인원이 맡을 연속 구간을 찾습니다.

#### DP 테이블 의미

```text
dp[z][end]
```

는 다음 의미를 가집니다.

```text
정렬된 주소 0번부터 end-1번까지를 z명에게 배정했을 때의 최소 비용
```

마지막 인원이 맡는 구간을 `start ~ end-1`로 두고, 이전 주소들은 `z-1`명이 이미 최적으로 나눴다고 가정합니다.

```text
dp[z][end] = min(dp[z - 1][start] + segmentCost(start, end))
```

#### 비용 함수

각 구간의 비용은 다음 요소를 반영합니다.

- 목표 오더 수보다 너무 적거나 많은 경우
- 목표 오더 수와의 차이
- 구간 내부에서 행정동이 바뀌는 횟수
- 같은 동 안에서 지번이 크게 튀는 정도

즉 단순히 오더 수만 맞추는 것이 아니라, 지리적으로 자연스러운 묶음을 만들도록 패널티를 부여합니다.

#### 결과 복원

DP 계산 후 `prev` 테이블을 역추적하여 실제 인원별 주소 구간을 복원합니다.

```text
인원 1: 주소 0 ~ 34
인원 2: 주소 35 ~ 71
인원 3: 주소 72 ~ 109
```

이 방식은 오더 균형과 주소 연속성을 동시에 고려할 수 있다는 장점이 있습니다.

## 지도 활용 방식(API)

### Naver Maps JavaScript API

지도 화면은 Naver Maps JavaScript API를 사용합니다.

활용 기능:

- `naver.maps.Map`: 지도 초기화
- `naver.maps.Marker`: 주소별 점 마커 표시
- `naver.maps.Polyline`: 배정 구역의 주소 이동 순서 표시
- `naver.maps.InfoWindow`: 주소 상세 정보 표시
- `naver.maps.LatLngBounds`: 센터/검색/인원 구역 화면 맞춤
- `naver.maps.CadastralLayer`: 지적도 레이어 표시

### Naver Geocoding API

관리자 페이지에서 엑셀을 업로드하면 서버가 Naver Geocoding API를 호출하여 주소를 좌표로 변환합니다.

```text
https://maps.apigw.ntruss.com/map-geocode/v2/geocode
```

## 프로젝트 구조

```text
assign_map/
├─ app.py                    # FastAPI 서버, 엑셀 처리, 지오코딩, 데이터셋 생성
├─ daejeon_map.html           # 지도 UI 및 배정 알고리즘
├─ admin.html                 # 관리자 업로드 화면
├─ process_geocode.json       # 지도 화면에서 사용하는 통합 데이터
├─ geocodes/
│  └─ geocoding.json          # 주소별 지오코딩 캐시
├─ uploaded_workbooks/        # 업로드된 원본 엑셀 파일
├─ api_keys.json              # Naver API 키 설정 파일
├─ requirements.txt           # Python 의존성
├─ run_assign_map.bat         # 실행 배치 파일
└─ assign_map.exe             # 배포용 실행 파일
```

## 실행 환경

### Python 실행

```bash
pip install -r requirements.txt
python app.py --bind 0.0.0.0 --port 8000 --open-browser
```

접속 주소:

```text
http://127.0.0.1:8000/
http://127.0.0.1:8000/admin
```

### 배치 파일 실행

```text
run_assign_map.bat
```

배치 파일은 Python 실행 환경이 있으면 최신 `app.py`를 실행하고, Python이 없으면 `assign_map.exe`를 실행하도록 구성되어 있습니다.

### exe 실행

Python이 설치되지 않은 사용자 PC에서는 `assign_map.exe`를 실행할 수 있습니다. 단, exe만 단독으로 복사하면 안 되고 `_internal`, HTML, JSON, geocodes 폴더 등 실행 폴더 전체가 함께 있어야 합니다.

## 프로젝트 한계

### 1. 주소 품질 의존성

원본 주소가 불완전하거나 지번 형식이 일관되지 않으면 지오코딩 정확도가 낮아질 수 있습니다.

### 2. API 호출 제한

Naver Geocoding API 호출량, 응답 속도, 네트워크 상태에 따라 관리자 갱신 시간이 달라질 수 있습니다.

### 3. 브라우저 렌더링 성능

주소 수가 많아질수록 지도 마커와 목록 렌더링 비용이 증가합니다. 특히 모든 센터와 모든 주소를 한 번에 표시하면 브라우저 성능의 영향을 받습니다.

### 4. DP 알고리즘의 범위

현재 배정 알고리즘은 정렬된 주소 목록을 연속 구간으로 나누는 방식입니다. 따라서 전체 공간에서 자유롭게 군집을 만드는 전역 최적화 문제와는 다릅니다.

### 5. 실제 이동 경로 미반영

현재 거리는 위도·경도 기반 직선거리 또는 정렬 기준을 활용합니다. 실제 차량 이동 시간, 도로망, 교통 상황은 반영하지 않습니다.

## 보안 및 운영 주의사항

- `api_keys.json`에는 실제 Naver API 키가 들어갈 수 있으므로 외부 저장소에 공개하지 않습니다.
- `uploaded_workbooks`에는 업무 원본 데이터가 포함될 수 있으므로 접근 권한을 제한해야 합니다.
- `process_geocode.json`, `geocoding.json`에도 주소와 좌표가 포함되므로 배포 범위를 관리해야 합니다.
- 공유 폴더에서 여러 사용자가 동시에 관리자 업로드를 실행하면 데이터 갱신 충돌이 발생할 수 있습니다.
