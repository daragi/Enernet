# 서비스오더 · 오생성(Error) 자동 필터링

> Enernet Center의 하위 기능으로, 서비스오더의 생성 소분류와 내역 텍스트 불일치를 검토·확정·분석하는 업무 자동화 시스템입니다.

운영 화면:

- Home: http://desktop-004:8000/
- 서비스오더 대시보드: http://desktop-004:8000/service_order/
- 오생성 관리자: http://desktop-004:8000/service_order/admin/

## 대시보드 전체 화면

현재 자동 캡처 연결을 사용할 수 없어 실제 운영 화면 이미지는 추가 대기 상태입니다. 개인정보가 마스킹된 대시보드 전체 화면을 확보하면 이 위치에 넣습니다.

권장 경로: `service_order/error/docs/images/service-order-dashboard.png`

## 1. 프로젝트 오버뷰

### 1.1 소개

서비스오더 원본 Excel을 업로드하면 전처리, 소분류별 키워드 매칭, 후보·자동 오생성 분류, 집계 저장, 대시보드 갱신을 한 번에 수행합니다. 관리자가 후보를 승인하거나 확정 오생성을 후보군으로 되돌리면 관리자 화면과 대시보드가 실시간으로 같은 상태를 사용합니다.

### 1.2 개요

시스템은 다음 네 가지 기능을 하나의 서버로 연결합니다.

1. 원본 Excel 업로드 및 전처리
2. 정상 규칙·충돌 규칙·오생성 규칙에 따른 분류
3. 관리자 후보 검토와 승인·롤백·제외
4. 사업부·소분류·인원·기간별 BI 대시보드

원본 전체 행과 후보 상세 데이터는 운영 화면의 현재 분석 작업으로 관리하고, 장기 집계와 승인 근거는 SQLite에 저장합니다. 과거 데이터도 같은 집계 기준으로 적재해 현재 데이터와 비교할 수 있습니다.

### 1.3 목표

- 생성부서와 상태 기준을 일관되게 적용해 지표의 분모를 고정
- 반복적인 정상 문구를 후보에서 제외해 수기 검토량 감소
- 확정 오생성의 재발을 규칙으로 누적해 자동 탐지 비중 확대
- 승인·롤백·재업로드 후에도 관리자 화면, Excel, 대시보드의 수치 정합성 유지
- 주소·전화번호·성명 등 개인정보가 결과 파일과 화면에 노출되지 않도록 보호

## 2. 문제 정의

### 2.1 기존 문제 상황

서비스오더 내역은 사람이 직접 입력하므로 같은 업무도 띄어쓰기, 특수기호, 날짜·금액, 고객 정보에 따라 표현이 달라집니다. 생성된 소분류와 내역이 맞지 않는 오생성을 단순 문자열 완전 일치만으로 잡으면 신규 표현을 놓치고, 충돌 키워드를 무작정 적용하면 후보가 폭증합니다.

또한 원본 집계와 검토 대상 집계를 섞으면 오생성률의 분모가 달라지고, 승인·롤백 시 관리자 목록과 대시보드가 서로 다른 숫자를 보여줄 수 있습니다.

### 2.2 비즈니스 질문 구조화

#### 2.2.1 우선순위형 질문

- 오생성이 가장 많이 발생한 사업부·소분류·인원은 어디인가?
- 지금 먼저 검토해야 하는 후보는 무엇인가?
- 반복적으로 재발한 내역 패턴은 무엇인가?

#### 2.2.2 상태확인형 질문

- 마지막 데이터가 어느 날짜까지 반영됐는가?
- 현재 후보군과 확정 오생성은 각각 몇 건인가?
- 승인·제외·롤백 결과가 대시보드에 반영됐는가?

#### 2.2.3 비교판단형 질문

- 전월 대비 오생성이 증가·감소한 사업부는 어디인가?
- 같은 기간 사업부별 오생성률 차이는 어떤가?
- 신규 패턴과 반복 패턴 중 어떤 유형이 늘었는가?

### 2.3 BI 표현 목표

첫 화면에서 마지막 업데이트 날짜, 전체 데이터, 오생성 건수, 오생성률을 확인하고, 기간 필터를 바꾸면 추이·사업부 현황·증감 원인·소분류 순위·인원 순위가 함께 바뀌도록 구성합니다. 숫자만 보여주는 대신 그래프 호버, 필터, 다운로드를 통해 원본 행까지 확인할 수 있게 합니다.

service_order/data/의 SQLite와 상태 파일, sap_id.json의 직원 매핑은 로컬 운영 데이터 또는 개인정보가 포함될 수 있어 Git에 올리지 않습니다. 원본 Excel, PKL, 업로드 임시 파일, 결과 임시 파일도 저장소에 포함하지 않습니다.

## 4. 데이터 전처리 과정

관리자에서 .xlsx를 업로드하면 다음 순서로 처리합니다.

1. 헤더 행과 열 이름을 인식하고 날짜·시간·숫자 형식을 통일합니다.
2. 서비스처리센터 관련 열을 정리하고 고객서비스처리센터를 서비스처리센터로 통일합니다.
3. 서비스처리센터 H051은 H073으로 병합합니다. 사업부 매핑은 센터 매핑과 CSC70, CSC7013 예외 규칙을 함께 적용합니다.
4. 분석 입력은 생성부서=사업부를 기본 범위로 사용하고, 상태=오더취소 행은 제외합니다.
5. 시스템에서 관리하는 예외 소분류·예외 행은 except_list.json 기준으로 처리합니다. 대시보드 표시 분모는 사업부 생성 건 중 오더취소가 아닌 행으로 고정합니다.
6. 내역 텍스트를 정규화합니다. 불필요한 공백·특수기호를 정리하고 날짜·금액·전화번호·주소·성명처럼 변동성이 큰 값은 패턴 매칭에서 일반화합니다.
7. 주소, 구주소, 전화번호, 성명 등 개인정보는 화면·후보 Excel·승인 상세 저장 시 마스킹합니다.
8. 전처리 결과와 후보·자동 오생성 결과를 원본 형식과 동일한 열 순서·스타일로 Excel에 저장하고 눈금선은 유지합니다.

업로드가 실패하면 이전 성공 작업을 유지합니다. 분석이 완료된 경우에만 새 작업을 현재 작업으로 교체합니다.

## 5. 키워드 매핑 과정

키워드는 전체 문장 하나를 그대로 비교하지 않고 소분류 범위 안에서 정규화된 토큰과 문구 조합을 비교합니다.

- normal_rules.json: 소분류별 정상 단일 키워드, 2어절·3어절 조합, 비연속 조합, 반복 문장 규칙
- error_rules.json: 관리자 승인 또는 과거 검증으로 확정된 오생성 패턴과 근거
- except_list.json: 자동 분류나 표시 집계에서 보호해야 하는 예외
- sap_id.json: 원본의 생성인 코드와 이름을 연결하는 로컬 전용 매핑

정상 규칙은 소분류별로만 적용합니다. 예를 들어 계량기일반의 대용량계량기교체는 다른 소분류의 정상 문구로 사용하지 않습니다. 패턴은 원문, 정규화 문구, 토큰, 적용 소분류, 역할과 출처를 함께 보존해 왜 제외됐는지 추적할 수 있게 합니다.

## 6. 충돌 로직 및 후보·자동 오생성 필터

판정의 기본 우선순위는 다음과 같습니다.

1. 생성된 소분류의 강한 정상 문구를 먼저 확인합니다.
2. 정상 문구가 여러 소분류에 걸쳐 발견되면 현재 소분류 우선 규칙과 충돌 보호 규칙을 적용합니다.
3. 정상으로 안정적으로 설명되지 않거나 다른 소분류 문구만 발견되면 후보군에 남깁니다.
4. 과거 확정 오생성 규칙과 높은 신뢰도로 일치하고 정상 규칙과 충돌하지 않는 경우에만 자동 오생성으로 승격합니다.
5. 애매한 충돌, 보호 규칙 일치, 정상·오생성 유사도가 비슷한 행은 자동 확정하지 않고 후보로 유지합니다.

따라서 자동 오생성은 재현율보다 정밀도를 우선한 보수적 영역이고, 후보군은 사람이 최종 판단하는 안전망입니다. 관리자가 후보를 승인하면 오생성 승인 상세와 날짜·생성인·사업부·소분류가 저장됩니다. 관리자가 후보로 되돌리거나 롤백하면 집계와 화면에서 즉시 빠지고, 재검토 가능한 후보 상태로 돌아갑니다.

## 7. 대시보드 구조

### Home

서비스오더 기능 진입점과 서버 연결 상태를 보여줍니다. 불필요한 업무 수치는 넣지 않고 브랜드·기능 중심으로 구성합니다.

### 서비스오더 대시보드

- 조회 기준: 인원·사업부, 월·기간, 연도·일자 필터
- 상단 현황: 마지막 데이터 날짜, 전체 데이터, 오생성 건수, 오생성률
- 오생성 추이: 일별 시계열 또는 누적 막대 전환
- 사업부 현황: 사업부별 건수·오생성률·전월 대비 증감
- 전월 대비 오생성 변화 분석: 사업부·소분류·인원별 증감
- 반복·신규 패턴: 기간 내 반복 패턴과 신규 패턴
- 소분류·인원 순위: 건수와 오생성률 전환
- 신규 오생성 모니터링: 목록 확인 및 Excel 다운로드

모든 주요 조회는 동일한 기간·사업부·인원 필터를 공유합니다. 관리자 승인·제외·롤백 이벤트가 발생하면 SSE와 브라우저 저장소 알림으로 대시보드 전체를 다시 조회합니다.

### 오생성 관리자

원본 업로드, 분석 진행률, 전처리 결과 다운로드, 후보군·오생성 탭, 열별 다중 필터, 후보 승인, 후보 복귀, 최근 승인 롤백, 신규 오생성 목록을 제공합니다.

## 8. 데이터 갱신·키워드 축적·학습 구조

### 데이터 갱신

1. 관리자가 최신 원본 Excel을 업로드합니다.
2. 서버가 업로드 파일에서 기간을 추출하고 전처리·후보 분류를 수행합니다.
3. 같은 기간을 다시 업로드하면 해당 날짜 구간의 집계를 삭제한 뒤 최신 결과로 교체합니다. 누적 합산하지 않습니다.
4. 기존 수기 승인 오생성은 오더번호 기준으로 다시 연결하고, 자동 오생성은 최신 파일 결과로 교체합니다.
5. 현재 분석 작업과 대시보드 집계가 커밋된 뒤 관리자·대시보드에 반영됩니다.

### 키워드 축적

- 후보 승인: 오생성 근거가 승인 저장소에 누적됩니다.
- 승인된 오생성: 중복 오더를 제거하고 error_rules.json 생성 근거에 반영됩니다.
- 반복 정상 문구: 안정적으로 정상 판정된 데이터에서 날짜·생성인·서로 다른 문장 조건을 확인한 뒤 normal_rules.json의 제안·활성 규칙으로 승격합니다.
- 규칙 변경 후에는 규칙 레지스트리를 재생성하고 현재 작업을 다시 분류합니다.

이 과정은 모델을 재학습하는 방식이 아니라, 운영자가 승인한 근거를 검증된 규칙으로 축적하는 방식입니다. 초기에는 후보 검토 비중이 높고, 반복 데이터가 쌓일수록 정상 문구 제외와 과거 오생성 자동 매칭이 늘어나는 구조입니다.

### 저장 정책

- 장기 보존: SQLite 집계, 승인 상세, 제외·롤백 상태, 과거 집계 상태
- 현재 작업: 임시 디렉터리의 최신 전처리·후보 Excel 및 현재 작업 manifest
- 미보존: 업로드 원본, 이전 작업의 전체 후보 행, PKL 캐시, 관리자 화면에 표시된 개인정보 원문

## 9. 주요 인사이트

현재 시스템에서 바로 확인할 수 있는 인사이트는 다음과 같습니다.

- 사업부별 오생성 건수와 오생성률의 차이
- 날짜별 오생성 집중 시점과 전월 대비 증감
- 반복적으로 재발하는 소분류·내역 패턴
- 새로 등장한 표현과 아직 규칙화되지 않은 후보
- 특정 소분류 또는 생성인에 집중된 오생성 위험
- 전체 데이터 대비 확정 오생성 비율과 후보 잔량

오생성 수치만 보지 않고 건수, 오생성률, 전체 데이터 분모, 기간, 사업부, 소분류를 함께 확인해야 업무량과 위험도를 올바르게 비교할 수 있습니다.

```powershell
cd C:\DA\center

```

### Git에 올리지 않는 파일

center/.gitignore에 따라 다음은 운영 컴퓨터에만 둡니다.

- service_order/data/의 SQLite·상태 파일
- service_order/error/json/sap_id.json의 직원명 매핑
- 원본 Excel, PKL, 업로드·결과 임시 파일

규칙 JSON과 소스 코드에는 원본 행 전체나 주소·전화번호를 저장하지 않습니다. 공개 원격 저장소에 올리기 전에는 개인정보·원본 파일·운영 DB가 git status에 나타나지 않는지 확인해야 합니다.
